# TARGET_PRODUCT_SPEC.md

## 1. Что именно строим

Итоговый продукт — это веб-сервис для автоматического распознавания координат цифр на чертежах.

На входе:
- один чертёж;
- или пара `чертёж + таблица`;
- или ZIP с набором пар, где связка идёт по имени файла.

На выходе:
- `CSV` с координатами цифр;
- `overlay`-превью;
- возможность открыть превью, проверить результат и вручную исправить ошибки.

Главная цель:
- находить координаты самого номера на чертеже;
- обязательно хранить `center` и `top_left`;
- в будущем использовать эти координаты для кликабельных зон на сайте.

## 2. Что входит в MVP

### 2.1. Основной сценарий

Пользователь:
- загружает один файл, пару файлов или ZIP;
- запускает обработку;
- ждёт фоновую обработку;
- получает экран результата;
- может скачать итоговый `CSV`;
- может открыть `overlay` и при необходимости исправить ошибки;
- после правки получает обновлённый финальный экспорт.

### 2.2. Поддерживаемые входы

В MVP входят:
- изображения;
- сканы;
- PDF;
- многостраничные PDF;
- режим без таблицы;
- режим с таблицей;
- ZIP с пакетной загрузкой.

Правила:
- если таблица есть, итоговый `CSV` должен содержать все номера из таблицы;
- если номер не найден, строка остаётся, но координаты пустые;
- если таблицы нет, система сама пытается найти все номера, которые реально относятся к чертежу.

### 2.3. Что должна делать система

Система должна:
- распознавать номер;
- определять `center`;
- определять `top_left`;
- отсекать ложные цифры, не относящиеся к чертежу:
  - номер листа;
  - цифры в логотипе;
  - служебные подписи вне смыслового контекста схемы;
- считать уверенность по результату;
- явно говорить, если не справилась надёжно.

Правило авторазметки:
- если уверенность по кандидату ниже условного порога около `70%`, координаты лучше не ставить автоматически.

### 2.4. Что должно быть на экране результата

На экране результата в MVP должны быть:
- статус обработки;
- итоговая оценка качества/уверенности по чертежу;
- сообщение, если результат ненадёжный;
- кнопка скачивания `CSV`;
- кнопка открытия `overlay`;
- минимальный ручной редактор.

Минимальный ручной редактор:
- сдвинуть точку;
- добавить точку;
- удалить точку;
- поменять label.

После ручной правки:
- `CSV` и `overlay` должны пересчитываться и становиться финальными.

## 3. Что не входит в MVP

Сознательно не включаем в MVP:
- многопользовательскую совместную работу;
- пользовательские сессии как отдельную сущность продукта;
- сложную историю действий и полноценный undo/redo как центральную функцию;
- детальную внутреннюю объяснимость пайплайна для пользователя;
- экспорт во множество форматов сверх реально нужных `CSV + overlay`;
- сложную продуктовую аналитику;
- кабинет администратора;
- полноценную каталоговую интеграцию с сайтом товаров.

Отдельно:
- уведомление «даже при закрытой вкладке» не стоит считать самым первым MVP-слоем;
- это требует отдельной реализации через браузерные push-уведомления / service worker и лучше делать следующим этапом после базового тестового продукта.

## 4. Какую основу брать

### Рекомендуемая база

Основу лучше брать из:
- `C:/projects/sites/blueprint-rec`

А из:
- `C:/projects/sites/blueprint-rec-2`

брать только отдельные идеи для preview/manual-review.

### Почему

`blueprint-rec` ближе к целевой модели продукта, потому что:
- думает пакетной обработкой;
- уже ориентирован на получение результата и экспорт;
- сильнее как база для `input -> pipeline -> output`;
- ближе к сценарию “пользователь загрузил и получил готовую таблицу”.

`blueprint-rec-2` полезен, но как вторичный источник:
- там сильнее интерактивный workspace;
- есть хорошие идеи для ручной проверки и правок;
- но его session-first модель тяжелее, чем нужно для целевого продукта.

Итоговое решение:
- ядро распознавания и batch-обработки — по логике `blueprint-rec`;
- слой preview и ручной правки — по мотивам `blueprint-rec-2`, но без переноса всей его архитектурной тяжести.

## 5. Как должен выглядеть тестовый продукт

Тестовый продукт должен уже уметь пройти сценарий от начала до конца:

1. Пользователь загружает:
   - один чертёж;
   - или пару `чертёж + таблица`;
   - или ZIP с несколькими парами.
2. Сервис ставит задачу в очередь и обрабатывает её в фоне.
3. Пользователь видит статус: `в очереди / в работе / готово / проблема`.
4. После завершения пользователь открывает экран результата.
5. На экране видно:
   - общую уверенность;
   - проблемные места;
   - `overlay`;
   - итоговый `CSV`.
6. Пользователь при необходимости правит ошибочные точки.
7. Сервис сохраняет правки и пересобирает финальный результат.
8. Пользователь скачивает финальный `CSV`.

## 6. Порядок этапов разработки

### Этап 1. Зафиксировать результат и формат данных

Сначала нужно жёстко зафиксировать:
- структуру входов;
- правила матчинга пар по именам;
- формат итогового `CSV`;
- правила для ненайденных номеров;
- единый формат координат в пикселях исходного изображения.

Цель этапа:
- убрать двусмысленность в том, что именно считается правильным результатом.

### Этап 2. Собрать сильное extraction-ядро

Нужно довести распознающий pipeline до состояния, где он:
- работает на одиночных файлах и пакетах;
- понимает PDF и многостраничность;
- даёт не только координаты, но и общую уверенность;
- умеет отсекать цифры вне контекста чертежа.

Цель этапа:
- получить надёжное ядро без привязки к сложному UI.

### Этап 3. Добавить честную оценку качества

Нужно ввести:
- score по кандидату;
- score по всему чертежу;
- флаг “результат ненадёжный”;
- список причин, почему результат сомнителен.

Цель этапа:
- чтобы сервис умел не только отвечать, но и признавать неуверенность.

### Этап 4. Сделать простой web-flow

Нужен минимальный веб-продукт:
- upload;
- очередь;
- статус;
- экран результата;
- скачивание `CSV` и `overlay`.

Цель этапа:
- довести путь пользователя до рабочего состояния без тяжёлого canvas-first интерфейса.

### Этап 5. Добавить manual review

После того как автообработка уже стабильна, добавляется preview-редактор:
- сдвиг;
- добавление;
- удаление;
- смена label;
- пересборка финального результата.

Цель этапа:
- закрыть оставшиеся ошибки без превращения продукта в ручной инструмент.

### Этап 6. Добавить batch UX и уведомления

После одиночного сценария:
- улучшить пакетную загрузку;
- показать прогресс по нескольким файлам;
- добавить звук;
- потом отдельным шагом решить уведомления при закрытой вкладке.

Цель этапа:
- довести сервис до реального рабочего сценария команды.

## 7. Что считать успехом MVP

MVP можно считать удачным, если:
- он стабильно принимает один файл, пару файлов и ZIP;
- сам запускает обработку в фоне;
- выдаёт `CSV` с правильной структурой;
- показывает overlay;
- честно показывает уверенность;
- позволяет исправить ошибки руками;
- после правок пересобирает финальный результат;
- не заставляет пользователя “разбираться в пайплайне”.

## 8. Главный продуктовый принцип

Не строить “умный лабиринт” вокруг AI.

Строить сервис, который:
- по умолчанию делает всё сам;
- если не уверен — честно говорит об этом;
- если ошибся — даёт быстро поправить;
- и в итоге всегда возвращает понятный и пригодный для бизнеса результат.
